# 🧠 Implementation Guide — Diagonal Rainwater Trapping

---

## 🟢 A. ELI5 Understanding

- Imagine each cell is a small pit.

- Rain falls everywhere.

- Water wants to **escape downhill**, but:
- It can ONLY move diagonally (↙ ↘)
- If it reaches the edge → it escapes

- So:

- 👉 If a cell has a path to boundary → NO water stays  
- 👉 If it is trapped → water accumulates

---

## 🔵 B. Plain-English Core Questions

### What information do I need?
- - Height of each cell
- - Whether water can escape from that cell

### What changes over time?
- - Cells get marked as “can escape”

### What must never change?
- - Movement is ONLY diagonal

### What exactly am I returning?
- - Total units of trapped water

### When do I discard past information?
- - Once a cell is marked “escape”, we never revisit

### What guarantees correctness?
- - If ANY path to boundary exists → no water

### Why avoid brute force?
- - Checking all paths per cell = exponential

### What breaks if constraints change?
- - If movement allowed in 4 directions → whole logic changes

---

## 🟣 C. Mathematical Modeling & Algorithm Derivation

### 1️⃣ Raw Reality Observation

- - Water escapes through paths
- - Paths are restricted to:
-   - (i+1, j-1)
-   - (i+1, j+1)

---

### 2️⃣ Ideal Model

- Instead of:
- > “Can water go OUT?”

- We ask:
- > “Can boundary reach this cell?”

- 👉 Reverse thinking

---

### 3️⃣ Structural Birth

- Let:

$$ escape[i][j] = \text{true if reachable from boundary via diagonal reverse paths} $$

---

### 4️⃣ Core Invariant

**Correctness means:**
- If a cell is marked reachable → water escapes

**Must always remain true:**
- Every visited cell has a valid escape path

**Failure looks like:**
- Marking a cell as trapped even though a path exists

---

### 5️⃣ Why This Works

- - Forward problem = many paths
- - Reverse problem = single BFS

- 👉 Converts exponential → linear

---

### 6️⃣ What Breaks If Altered

- - If we allow horizontal moves → incorrect marking
- - If we don't start from boundary → incomplete detection

---

## 🟠 D. High-Level Explanation

- 1. Start from all boundary cells  
- 2. Perform BFS in reverse diagonal directions  
- 3. Mark all reachable cells as “escape”  
- 4. Remaining cells are trapped  
- 5. Compute water using neighbor height constraints  

---

## 🟡 E. Detailed Logic Breakdown

**Step 1: Initialize visited matrix**
- - Marks escape cells

**Step 2: Add boundary cells**
- - All edges → starting points

**Step 3: BFS traversal**
- - Move only:
-   - up-left
-   - up-right

**Step 4: Condition**
- - Only move if next height ≤ current height

**Step 5: Mark reachable**
- - These cells cannot trap water

**Step 6: Compute trapped water**
- - For unvisited cells:
-   - Check both diagonal children
-   - Take minimum boundary

---

## 🧱 F. Boundary & Guard Logic

**Case: Empty grid**
- - Return 0 immediately

**Case: Single row/column**
- - No diagonal containment → return 0

**Case: Out of bounds**
- - Always check before accessing

**Case: No valid neighbors**
- - Cannot form boundary → no water
